# Product Requirements Document: Bastion Agentic Defense System SIEM Expansion

## 1. Executive Summary

Bastion transforms from a Solana native AI agent transaction firewall into a blockchain native Security Information and Event Management platform. The upgraded system writes every security decision to the Solana blockchain as a verifiable and tamper proof audit trail. Daemon Blockint Technologies serves as the intelligence source of truth powering the blockint rules engine, the GrondOSINT threat oracle, and the 47 agent skills ecosystem. EVM chain support including Celo and Base is deferred to a future development phase so that all engineering resources focus exclusively on Solana. The platform provides the first SIEM where the audit trail itself is the product, giving Indonesian regulators, financial institutions, and enterprises a sovereign and verifiable evidence layer that no foreign SIEM vendor can replicate.

## 2. The Daemon Ecosystem

Daemon Blockint Technologies provides three core assets that power Bastion. First is GrondOSINT, an agentic OSINT platform querying Tavily for web search, Shodan for internet infrastructure, and Twitter for social signals, feeding real world threat data into Bastion's risk oracle. Second is the Blockint rules engine detecting flash loan patterns, high slippage trades, mint authority changes, freeze authority changes, and risk labeled blockchain addresses. Third is the agent skills ecosystem of 47 specialized skills covering blockchain forensics, compliance workflows, DeFi security auditing, MEV investigation, rug pull detection, and cross chain clustering. These skills are distributed through the agentic reserve blockint skills repository on GitHub and locked by SHA-256 hashes in Bastion's skills lock file. Grond injection skills feed threat intelligence directly into the correlation engine, enabling real time enrichment of security events with address reputation data, entity labeling, and risk scoring before events reach the policy evaluation layer.

## 3. Product Vision

The current Bastion v0.3.0 is a multichain AI agent firewall intercepting Solana transactions, simulating them against live chain state via Helius, enforcing blockint security rules, and recording decisions on chain. The SIEM expansion lifts this architecture from blockchain transactions to any security event stream. Any source such as cloud audit logs, API access events, authentication flows, network telemetry, and smart contract events flows through the same pipeline of ingestion, policy evaluation, threat correlation, and on chain audit recording. The result is a unified security operations console where SOC analysts triage alerts, investigate incidents, and produce verifiable evidence for regulatory compliance. The system treats every event as an immutable record anchored to Solana.

## 4. MCP Integration and Agent Interface

The Model Context Protocol server exposes Bastion's security capabilities to AI agents including Claude, Cursor, and Codex. The private bastion mcp repository implements five tools: simulate transaction, retrieve audit logs, register agent identity, override blocked transactions, and query policy configuration. Two prompts are provided for agent workflows: verify transaction safety before signing, and review audit history for compliance reporting. The MCP server supports both standard input and output transport for local agents and Server Sent Events transport for remote agents. Payment for API access is enforced through the x402 protocol using Solana program derived token accounts. Agent platform integration includes a Claude Code marketplace plugin, Cursor rules configuration, Codex AGENTS.md instructions with inline SDK examples, and OpenClaw hub installation. A Netlify edge function already serves the dashboard documentation as structured markdown to AI agents via content negotiation on the Accept header. The public MCP server card at the well-known endpoint exposes simulate_transaction, check_policy, and get_audit_logs tools for discovery by any MCP client.

## 5. SDK Architecture

The public TypeScript SDK exports the BastionClient class with 15 methods covering all 7 Anchor program instructions plus account fetching, event listening, and PDA derivation. Agent capabilities are encoded as a bitmap constant supporting transfer, swap, NFT mint, NFT transfer, stake, delegate, and create program operations. The decision enum defines three states: allowed, blocked, and pending. A BastionX402Client extension adds automatic payment retry logic with on chain transaction verification on Solana. The SDK re-exports all types from types.ts including AuditState, AuditEntry, Agent, Policy, and the AGENT_CAPABILITIES and DECISION constants. All methods return Promise of Transaction, deferring signing and sending to the caller. Pricing tiers are free for 100 requests per month, builder at 0.01 SOL per request, pro at 0.005 SOL per request, and enterprise with custom pricing. The universal ingest endpoint will be exposed as a new client method accepting any SecurityEvent JSON payload with automatic normalization and routing through the policy engine.

## 6. Current SIEM Capabilities

Event ingestion is handled by the Axum HTTP interceptor exposing the simulate endpoint for transaction validation and the evaluate v2 endpoint for chain agnostic policy evaluation. The policy engine enforces program allowlists, Solana native token caps, per minute rate limits, and all five blockint security rules. Audit logging operates at two layers: a local Sled embedded database for fast querying and an Anchor on chain program writing immutable audit entries to Solana devnet at program ID BaSZuLcwjfh75T3TjbVYpTH4qpJt1tNoZ3S6PTkvNhCb. The human override dashboard presents a manual approval queue where security operators can review blocked transactions and approve or reject them with one click. The circuit breaker provides fleet wide transaction pausing with on chain pause and resume instructions that trigger the Anchor program's emergency_pause and emergency_resume methods. GrondOSINT provides the threat intelligence feed scoring blockchain addresses for risk. The sidecar API exposes paginated audit logs, policy CRUD, circuit breaker status, and health monitoring endpoints.

## 7. Gap Analysis and Missing Capabilities

Four major capabilities are missing for full SIEM functionality. The correlation engine does not exist yet: incoming events are evaluated individually without cross event pattern matching. Ingestion is limited to Solana transactions only: there is no universal endpoint for arbitrary event formats such as syslog, CloudTrail, or authentication logs. Threat scoring and MITRE ATT&CK mapping are not implemented: blocked events are not classified by threat type or severity. Case management is absent: the dashboard has no workflow for promoting blocked events into formal investigation cases with analyst assignment and resolution tracking. All four gaps have clear mapping to the existing architecture, meaning each can be implemented as an additive module without rewriting core components.

## 8. Phase 1: Universal Multi Source Ingestion

A new universal ingest endpoint will be added to the Axum HTTP server accepting Common Event Format data and custom JSON schemas. Adaptor modules will translate external event sources into normalized security events: AWS CloudTrail for infrastructure audit logs, GitHub webhooks for code repository events, Slack audit logs for communication platform monitoring, syslog for network device telemetry, and OJK compliant financial event streams for Indonesian regulatory reporting. The existing NormalizedTransaction type in the chain agnostic core crate will be generalized into a SecurityEvent type supporting arbitrary event sources while preserving blockchain transaction fields. An ontology manager inspired by Palantir Foundry patterns will structure event data into typed objects, properties, and links enabling complex queries across heterogeneous event sources. Every ingested event will receive a unique identifier, a timestamp in Unix seconds, a source label identifying the originating system, and a classification tag from a controlled vocabulary covering authentication, authorization, transaction, configuration, and network categories.

## 9. Phase 2: Correlation Engine and Threat Scoring

A new Rust crate called bastion correlation will implement the correlation engine. It maintains a sliding time window ring buffer of recent SecurityEvent objects configurable from one minute to twenty four hours. Correlation rules are defined in YAML files specifying sequences of events that constitute a threat pattern. One example rule detects three failed authentication attempts followed by a large balance transfer exceeding 5 SOL within a 60 second window, triggering a high severity alert. Another example rule detects a program upgrade transaction immediately preceded by an admin key rotation within 300 seconds, triggering a critical severity alert. When a rule matches, the engine produces a CorrelationAlert with a severity level of low, medium, high, or critical, a list of contributing event identifiers, and a human readable description of the detected pattern. The correlation engine integrates directly with GrondOSINT as its threat context layer, enriching alerts with real world intelligence about involved addresses and entities sourced from Daemon Blockint Technologies. Detected patterns are mapped to MITRE ATT&CK for ICS and the emerging MITRE ATT&CK for Web3 categories for standardized threat reporting. Both the correlation rules and the GrondOSINT threat data pipe directly from Daemon Blockint Technologies as the authoritative source of truth.

## 10. Phase 3: On Chain Audit as Compliance Primitive

Every security decision produced by the correlation engine is written to the Solana blockchain through the existing Anchor audit program. The on chain AuditEntry account stores the decision outcome as an 8 bit unsigned integer where 0 means allowed and 1 means blocked, the simulation result hash as 32 bytes, the reasoning text as a variable length UTF-8 string, and an optional program identifier as 32 bytes. This creates an immutable and publicly verifiable audit trail. For the Indonesian regulatory context, POJK 11 of 2022 requires financial institutions to maintain tamper proof audit logs. The Solana Anchor program satisfies this requirement natively because blockchain records cannot be altered after they are written. BSSN, the Indonesian national cybersecurity agency, can verify audit entries independently by querying the Solana blockchain through any public RPC endpoint including api.devnet.solana.com. The on chain audit trail also serves as admissible evidence in legal proceedings and regulatory examinations because it proves exactly what security decision was made, when it was made, and by which agent authority. Links between the case management module and on chain records create a bidirectional traceability chain from dashboard to blockchain.

## 11. Phase 4: Case Management Module

A case management panel will be added to the existing React and Vite dashboard application. A SOC analyst can promote any blocked event from the audit logs table into a formal investigation case with one click. The case captures a unique case identifier using UUID version 4, the analyst assigned to investigate, the current status of open, in progress, resolved, or closed, a timeline of actions taken with timestamps, and a list of attached evidence items. Evidence items include on chain transaction hashes that serve as immutable proof links directly to Solana block explorers. The case timeline view correlates multiple related events into a single unified incident display using the correlation engine's alert grouping. An analyst can add notes in free text, change the status through a dropdown, and attach files such as screenshots or exported logs. When a case is resolved, the resolution summary and all evidence links are written to the Solana blockchain as a final case closure record through a new close_case Anchor instruction. The case index is stored in Sled DB for fast local querying while the closure evidence remains on chain.

## 12. Blockint Rules and Daemon Threat Intelligence

The five Blockint security rules that already exist in the Rust policy engine are expanded with additional rules from Daemon's intelligence pipeline. Flash loan detection flags transactions where the borrow and repay amounts are nearly equal within the same atomic transaction, identified by matching token inflow to token outflow. High slippage detection blocks trades exceeding a configurable basis point threshold with a default of 500 basis points or 5 percent, comparing expected output against simulated output. Mint authority change detection blocks any transaction attempting to modify the mint authority of a token through the SPL Token program. Freeze authority change detection blocks any transaction attempting to modify the freeze authority of a token. Risk labeled address screening blocks transactions involving addresses flagged in the GrondOSINT database with a configurable minimum confidence threshold. New rules to be added include contract upgrade detection for proxy contracts by monitoring the program upgrade authority instruction, admin key rotation monitoring by tracking changes to program upgrade authorities, liquidity removal detection by identifying large LP token burns without corresponding swaps, and oracle manipulation pattern recognition by comparing on chain oracle prices against off chain reference prices from GrondOSINT sources. All rules execute before the Helius simulation so that transactions are rejected at the policy layer without consuming simulation compute units.

## 14. W3C DID Compliance Architecture

Bastion will adopt the W3C Decentralized Identifiers specification version 1.0 to create a unified identity layer spanning Solana, EVM chains, and Midnight Network. Every agent registered on any chain receives a did:bastion identifier that resolves to a W3C compliant DID document. The DID method is defined as did:bastion followed by a chain-specific namespace and the agent identifier, for example did:bastion:solana:BaSZuL and did:bastion:eip155:1:0xAbCd. The DID document contains the agent's Ed25519 or secp256k1 public keys as verification methods, the supported capabilities expressed as a service endpoint with the capability bitmap, the current reputation score as a numeric property, and service endpoints pointing to the Bastion sidecar API and MCP server for secure agent communication. The DID document is generated dynamically by a resolver service that queries the Solana PDA for the primary identity, the ERC-8004 token for EVM cross chain proofs, and the Midnight ZK commitment for privacy preserving identity verification. The resolver caches DID documents in Sled DB with configurable time to live and serves them through a standard REST endpoint at slash did slash resolve. This architecture aligns with W3C's DID Core specification, the DID Resolution specification, and supports JSON-LD representation for semantic interoperability with external identity systems.

## 15. EIP-8004 as the EVM Identity Anchor

The existing BastionERC8004Registry contract at evm slash src slash BastionERC8004Registry dot sol implements the full ERC-8004 Trustless Agents Identity Registry specification across 345 lines of Solidity. Each agent receives an ERC-721 soulbound token minted to the registering address. The token URI points to a registration file stored on IPFS or HTTPS containing the agent name, capabilities, and reputation metadata. The contract supports on chain metadata storage through the setMetadata and getMetadata functions using key value pairs with reserved key protection for sensitive fields such as agentWallet. The agent wallet binding uses EIP-712 typed structured data signatures for externally owned accounts and ERC-1271 magic value verification for smart contract accounts, with deadline enforcement to prevent replay attacks. Token transfers automatically clear the agent wallet binding to prevent unauthorized wallet carry over. The global agent identifier follows the CAIP-10 format as eip155 followed by the chain ID followed by the checksum address, for example eip155:42220:0x1234 for Celo mainnet. In the expanded DID architecture, the ERC-8004 token serves as the EVM anchor for the unified DID document through an alsoKnownAs field in the DID document pointing to the ERC-8004 global identifier. The resolver service verifies ERC-8004 token ownership by querying the contract's ownerOf function and cross referencing it with the DID controller field.

## 16. Midnight Agent DID Manager

A new component called the Midnight Agent DID Manager bridges Midnight Network ZK identities with the W3C DID standard. It runs as a Rust service integrated with the existing Midnight middleware at midnight slash middleware slash src. It maintains a mapping from Midnight ZK commitments stored as Field values in the registry compact to DID identifiers. When an agent registers on Midnight through the registerAgent circuit, the DID Manager creates a DID document with the ZK commitment as a verification method of type Ed25519VerificationKey2020 using the Jubjub curve native to Midnight. The DID document includes a service endpoint pointing to the Midnight middleware for selective disclosure requests. The service endpoint exposes a disclose DID verification method that accepts a viewer key and returns only the specific fields the agent has authorized for disclosure, powered by the discloseEntry circuit in audit dot compact. When an auditor needs to verify an agent identity, they present the DID to the disclosure circuit which reveals the agent's public key, name, and capabilities without exposing the transaction history, policy configurations, or target addresses. This provides General Data Protection Regulation compliant identity management where the agent controls exactly what information is disclosed and to whom. The DID Manager integrates with the existing Rust Axum middleware and the TypeScript SDK through a new resolve DID method that internally maps ZK commitments to DID documents.

## 17. Unified Identity Flow

An agent operator registers once on Solana through the SDK's registerAgent method, receiving a PDA account seeded by the bastion_agent string and the authority public key. This Solana PDA stores the agent name, capability bitmap, reputation score, and registration timestamp. The PDA account address becomes the primary identifier in the did:bastion:solana namespace. The same agent can then prove ownership of that Solana identity to mint an ERC-8004 token on any EVM chain by signing an EIP-712 message with the Solana wallet's corresponding secp256k1 key derived from the same seed phrase. The ERC-8004 token mints to the agent's EVM address and records the did:bastion:solana identifier in the token URI. On Midnight, the agent submits a ZK commitment derived by hashing the DID identifier with a secret nonce using the SHA256 based derivation in the Midnight SDK's proof module, registering without revealing the underlying Solana or EVM identity. The commitment maps to the same underlying DID document through the Midnight Agent DID Manager. All three identities link back to the same DID document, creating a unified cross chain agent identity system compliant with W3C DID, EIP-8004, and capable of ZK privacy through Midnight. An auditor can verify all three identity anchors independently and trace them to a single agent sovereign identity.

## 20. Competitive Analysis

Traditional SIEM platforms including Splunk, IBM QRadar, and Microsoft Sentinel store logs in mutable databases that administrators can delete or modify. Bastion writes every security decision to the Solana blockchain where it becomes an immutable and publicly verifiable record. This on chain audit trail is Bastion's singular competitive differentiator. No commercial SIEM vendor offers blockchain anchored evidence today. The W3C DID compliance provides sovereign identity management that no SIEM vendor offers, enabling cross chain agent identity with cryptographic verification. The Midnight ZK privacy layer provides selective disclosure that no SIEM vendor offers, enabling GDPR compliant audit trails where sensitive data remains private until disclosure is explicitly authorized. For the Indonesian market specifically, foreign SIEM vendors cannot provide sovereign data residency, cannot align with OJK tamper proof audit requirements without complex configuration, and cannot offer BSSN verifiable evidence chains. Bastion solves all three by running on the Solana blockchain which is globally accessible for independent verification while keeping the sidecar deployment within Indonesian jurisdiction. The GrondOSINT threat intelligence feed sourced from Daemon Blockint Technologies provides a competitive moat that commercial SIEM vendors cannot replicate because it combines OSINT, blockchain forensics, and agentic AI threat detection in a single pipeline.
